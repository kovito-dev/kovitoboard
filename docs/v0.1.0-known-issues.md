# v0.1.0 既知の未解決課題

## 1. ビルド・配布

### 1-1. dist/ の ESM モジュール解決問題
- **概要**: `tsc` が出力する `dist/server/*.js` は拡張子なしの import（`import { foo } from './bar'`）を含む。Node.js の ESM ローダーは拡張子を要求するため、`node dist/server/index.js` では動作しない
- **現在の回避策**: `tsx` ランタイムで実行（`npm run start` = `tsx src/server/index.ts`）
- **影響**: 開発・動作には影響なし。npm publish 時に問題となる
- **対応方針**: v0.2.0 でサーバーバンドラー（esbuild 等）を導入し、単一 JS ファイルに出力する

### 1-2. フロントエンドのチャンクサイズ警告
- **概要**: `dist/assets/index-*.js` が 500KB を超える（約 626KB）
- **影響**: 初期ロードがわずかに遅い（ローカル動作のため実質的な影響は軽微）
- **対応方針**: v0.2.0 でコード分割（dynamic import）を検討

## 2. 機能制限

### 2-1. エージェント編集機能なし
- **概要**: v0.1.0 ではエージェント定義（`.claude/agents/*.md`）の閲覧のみ。作成・編集・削除は UI からできない
- **対応方針**: v0.2.0 で CRUD UI を追加

### 2-2. SettingsModal は読み取り専用
- **概要**: 設定画面（5タブ: 基本/スキル/自動処理/連携/ルール）は GET API のみで表示。編集・保存は不可
- **対応方針**: v0.2.0 で POST API と編集フォームを追加

### 2-3. trust-prompt の完全 E2E フロー未検証
- **概要**: tmux パネル監視 → パターン検出 → UI モーダル表示 → 選択肢応答 → tmux キー送信 の一連フローは、tmux セッションが稼働している環境でのみ検証可能
- **Phase 8 での確認結果**: API レベル（`/api/tmux/status`, WebSocket 接続, trust_prompt_respond メッセージ処理）は正常動作を確認済み。UI モーダルのレンダリングも E2E テストで確認済み（非表示状態）
- **対応方針**: v0.1.0 リリース前に tmux + Claude Code 環境での手動テストを実施

## 3. テスト

### 3-1. E2E テストのエージェント依存
- **概要**: agent-management と session-flow の一部テスト（2件）は `.claude/agents/` にエージェント定義がない環境ではスキップされる
- **影響**: CI 環境ではテストカバレッジが低下する
- **対応方針**: CI 用のダミーエージェント定義 fixture を用意するか、テスト内で一時ファイルを作成する方式に変更

### 3-2. IPv6/IPv4 ローカルホスト問題
- **概要**: Playwright の `request` fixture が `localhost` を IPv6（`::1`）に解決する環境がある。E2E テストでは `127.0.0.1` に統一して回避済み
- **影響**: テストコード内のハードコーディング。環境依存の問題が再発する可能性
- **対応方針**: サーバー側を `0.0.0.0` でリッスンするか、テストで環境変数から URL を取得する方式に変更

## 4. 設計上の負債

### 4-1. gala 由来の残存コード
- **概要**: Phase 2 でタスク機能は剥離済みだが、SettingsModal の 5 タブ構成（skills/automations/integrations/rules）は gala の設計をそのまま引き継いでいる。KovitoBoard の設計思想に合うかは未検討
- **対応方針**: v0.2.0 で architect が設定画面の仕様を再設計

### 4-2. upload API の残存
- **概要**: `POST /api/upload` が残っている。v0.1.0 では使用されていないが、削除もされていない
- **対応方針**: Phase 2 の判断で「Phase 3 で kovitoboard-uploads にリネーム」とされたが、実際にはリネームされていない。v0.2.0 で方針を再決定

### 4-3. `resolveProjectRoot` の推測ロジック脆弱性（DEC-009 で対応中）
- **概要**: `src/server/config.ts` の `resolveProjectRoot()` は `__dirname` 起点で階層数を推測し、`CLAUDE.md` の存在で判定している。以下の問題がある:
  - tsx 実行時 / dist 実行時の階層差をハードコード（bundler 導入や npm publish で破綻）
  - ユーザープロジェクトに `CLAUDE.md` が無い場合、candidates[0] に黙ってフォールバックして誤ったパスで `.kovitoboard/` を作成する
  - 「KB の設置場所」と「ユーザープロジェクトのルート」を暗黙に同一視している
- **影響**: 開発環境では概ね正しく動くが、誤ったパスで `.kovitoboard/` が作られるケースが散見される
- **対応方針**: **DEC-009（v0.1.0 採用方針）** に従って解決:
  - 短期（v0.1.0）: 優先順位 `--project-root` 引数 → `KOVITOBOARD_PROJECT_ROOT` env → `.kovitoboard/setting.json` の `project.path` → `process.cwd()`
  - オンボーディング Step 3 で `projectRoot` を表示し、完了時に `setting.json` に永続化
  - `__dirname` 推測と `CLAUDE.md` 判定は撤廃
- **参照**: `kovito-hq/docs/design/decisions/DEC-009-project-root-resolution.md`
