## この章の目的

レシピの仕組み（サンドボックス内で動く宣言的 handler モデル）を理解し、同梱サンプルレシピの有効化・実行、自作機能のレシピ出力、scope 承認の判断ができるようになることを目的とします。

> ⚠️ **v0.2.x の重要な前提（読み飛ばさないでください）:** **外部レシピのインストールは v0.2.x では無効です**（`/api/recipes/install` / `/api/recipes/apply` はいずれも 410 Gone）。レシピの受け取り・インストール経路は **v0.3.0 で KovitoHub（署名付き publisher モデル）と連動して提供予定**です。v0.2.x で利用できるのは **KB 同梱のサンプルレシピ**（Document Viewer / TODO）を「アプリ」画面の **Sample apps タブ**から有効化する経路と、既にインストール済みのレシピの閲覧・無効化・エクスポートのみです。エージェントは利用者に「外部レシピを今すぐインストールできる」と案内してはいけません（§2 参照）。

## §1 レシピとは

レシピは、KB に **機能の塊（画面 + バックエンド呼び出しの宣言）を追加できる、配布・受け取り可能な単位** です。ドキュメントビュアーや TODO 管理のような「1 つの目的を持ったミニアプリ」を、サンドボックス内で安全に追加できる設計です。（**注:** 外部レシピの受け取り・インストール経路は v0.2.x では無効で、v0.3.0 で KovitoHub と連動して提供予定です。v0.2.x で利用できるのは同梱サンプルレシピの有効化です。§2 参照）

### 1.1 レシピの構造（ユーザーから見た視点）

レシピには以下が含まれます:

- **レシピ ID（`recipeId`）:** レシピを一意に識別する不変識別子（配布者が決定。例: `document-viewer` / `kovito-dev/document-viewer`）。レシピ画面ではこの ID を直接見ることはほぼありませんが、内部的にインストール状態の管理やアプリの逆引きに使われます
- **成果物:** 実際に追加される UI ページ・スタイル（`app/pages/*.tsx` 等）の参考コード
- **メニュー登録:** サイドバーのどこに表示するか（初期アプリ ID 候補も含む）
- **BE 呼び出しの宣言（任意）:** ファイル読み取り・KV 保存等の「どの handler を呼ぶか」を YAML で宣言（§6 参照）
- **scope（権限）の宣言:** 何にアクセスしてよいか（§7 参照）
- **指示（任意）:** エージェントへの補足説明

**レシピ ID とアプリ ID の違い:**

| 項目 | recipeId | アプリ ID（appId） |
|---|---|---|
| 由来 | レシピ配布者が決定 | KovitoBoard ローカルでの命名（衝突時は自動でサフィックス採番） |
| 不変性 | 変わらない（レシピの identity） | 採番時に決まる、後から自動変更されない |
| 例 | `document-viewer` | `document-viewer`、または同名衝突時 `document-viewer-2` |

同じレシピを 2 回インストールすると、レシピ ID は同じだが、アプリ ID は 2 つ存在する（再インストールで別 appId として配置）。

### 1.2 レシピとアプリ（`app/`）の違い

| 項目 | レシピ | ユーザー定義のバックエンド API（`app/api/`） |
|---|---|---|
| 実行経路 | 宣言的 handler モデル経由（サンドボックス） | Node.js でフル権限実行 |
| 権限 | `scope` で宣言した範囲のみ | 制限なし |
| 信頼モデル | 「第三者から受け取っても安全に試せる」 | 「自分で書いたコード、自己責任」 |
| コード実行 | レシピは **コードを書かず宣言するだけ** | TypeScript を自由に書く |
| 反映方法 | インストール直後にブラウザを再読み込み（サーバー再起動は不要） | `app/api/*.ts` を変更したらサーバー再起動が必要 |

**判断基準:**

- 「他人に渡したい」「サンドボックスで安全に配布したい」 → **レシピ**（外部インストールは v0.3.0 予定、§2）
- 「自分専用の作り込んだ機能が欲しい」「フル権限で何でもしたい」 → [`05-apps.md`](./05-apps.md) の **`app/`** 配下で作る

### 1.3 v0.2.x でのスコープ

- レシピの FE 拡張（`page` 型の成果物）は正式サポート
- BE 拡張は **宣言的 handler モデル** で Category A（9 個、ファイル読み書き・KV・通知・エクスポート）のみ利用可能
- HTTP 外部通信・シェル実行等は未対応（v0.3.0+ で別判断）
- **外部レシピのインストール経路は v0.2.x では無効**（v0.3.0 で KovitoHub と連動して提供予定）。同梱サンプルレシピの有効化と、既存レシピの閲覧・無効化・エクスポートは利用可能（§2 参照）

## §2 外部レシピの読み込み（v0.2.x では無効・v0.3.0 で再開予定）

> 🛑 **エージェント向けの最重要注意:** 外部レシピを「今すぐインストールできる」と利用者に案内してはいけません。**v0.2.x ではインストール経路が無効**です。利用者が外部レシピを入れたいと相談してきたら、(a) v0.3.0 で KovitoHub と連動して提供予定であること、(b) 今すぐ試せるのは同梱サンプルレシピ（§5）であることを正確に伝えてください。

### 2.1 アプリ画面の「Recipes」タブ

外部レシピは「アプリ」画面の **Recipes タブ** に表示されます。v0.2.x ではこのタブに次の案内バナーが表示されます:

> **v0.3.0 で KovitoHub と連動して提供予定** — KovitoHub から署名済みレシピをインストールできます。各レシピは publisher により署名され、KovitoBoard が監査します。

つまり Recipes タブは v0.2.x では**将来機能の予告**であり、ここから外部レシピをインストールすることはできません。

### 2.2 v0.2.x でできること / できないこと

| 操作 | v0.2.x | 備考 |
|---|---|---|
| 外部レシピ（`.zip` / `.yaml` / URL）のインストール | ❌ 無効 | `/api/recipes/install` は 410 Gone。v0.3.0 で KovitoHub 経由で再開予定 |
| 旧 apply フロー | ❌ 撤去 | `/api/recipes/apply` も 410 Gone（v0.2.x で撤去） |
| 同梱サンプルレシピの有効化 | ✅ 可能 | 「アプリ」画面の **Sample apps タブ**から有効化（§5） |
| 既にインストール済みレシピの閲覧 | ✅ 可能 | インストール履歴・実行履歴の確認（§3） |
| 既存アプリの無効化（同梱サンプルのみ・非破壊）/ 削除 | ✅ 可能 | 同梱サンプルは Sample apps タブで非破壊に無効化、独自・レシピ由来アプリは削除のみ（[`05-apps.md`](./05-apps.md) §10） |
| 自作機能のレシピ出力（エクスポート） | ✅ 可能 | §4（`api/` を含むアプリは拒否、§4.3 参照） |

### 2.3 既存レシピの grandfather（移行措置）

v0.1.x / v0.2.0 で既にインストール済みのレシピ、および同梱サンプルレシピは、インストール経路の無効化後も次が維持されます:

- インストール履歴・実行履歴の閲覧（§3）
- 同梱サンプル由来アプリの**無効化（非破壊）** / 各種アプリの完全削除（[`05-apps.md`](./05-apps.md) §10）。非破壊の無効化は同梱サンプル限定で、その際 `app/data/<appId>/` 配下の利用者データは保持されます（独自・レシピ由来アプリは削除のみ）
- レシピとしての**エクスポート**（§4）

### 2.4 v0.3.0 で予定されているインストールモデル（参考）

外部レシピのインストールは v0.3.0 で **KovitoHub（署名付き publisher モデル）** と連動して再開する計画です。各レシピは publisher により署名され、KovitoBoard が監査したうえで配布される設計が検討されています。本章の旧「受け取り → エージェント対話 → 実装」フローは v0.3.0 の設計確定時に改めて記載します。それまでは外部レシピのインストールは提供されません。

## §3 レシピの実行と履歴

### 3.1 レシピの実行

インストール済みレシピは、レシピが追加したメニュー項目（§1.1）から使います。レシピ固有の画面が開き、そこで操作します。

### 3.2 履歴の確認

「インストール済みレシピ」の各カードから、以下が確認できます:

- **インストール日時・経路**（ファイル / URL / サンプル）
- **承認済み scope**
- **実行履歴**（いつどの handler を何回呼んだか、`app/data/{recipe-id}/_audit.log` に記録）

### 3.3 レシピの再インストールと削除

レシピ画面に「アンインストール」ボタンはありません。レシピが作ったアプリの削除は **アプリ画面の削除ボタン**（[`05-apps.md`](./05-apps.md) §10 参照）から実施します。

#### 3.3.1 レシピの再有効化・再インストール

- **同梱サンプルレシピ:** 「アプリ」画面の **Sample apps タブ**から、無効化したサンプルをいつでも再度有効化できます（§5）。無効化時に `app/data/<appId>/` は保持されるため、再有効化で従来データを引き継げます。
- **外部レシピの再インストール:** v0.2.x では外部レシピのインストール経路自体が無効のため、再インストールもできません（§2）。v0.3.0 の KovitoHub 連動で再開予定です。

#### 3.3.2 アプリの削除（旧アンインストール）

レシピが作ったアプリを完全に削除したい場合:

1. **アプリ画面を開く**（サイドバーから対象アプリを選択）
2. サイドバー上部の「**アプリを削除**」ボタンをクリック
3. 確認モーダル → エージェント選択 → エージェントが対話で削除を実施
4. 削除内容:
   - `app/<appId>/` 配下のコード一式
   - `app/menu.ts` からの該当エントリ
   - `app/data/<appId>/` 配下の専用データ領域（エージェントが対話で確認後）

**重要:** アプリ削除を実施しても、**レシピのインストール済バッジは消えません**。「1 度でもインストールしたレシピはインストール済」として履歴ベースで判定するため、再度同じレシピを使いたい場合は §3.3.1 の再インストールが可能です。

#### 3.3.3 データのバックアップ

エージェントは削除前に「`app/data/<appId>/` を削除してよいか」を再度確認します。業務データなど大切なデータが含まれる場合は、事前にバックアップしておくことを推奨します。

## §4 自作機能のレシピ出力

KB 上で自分が作った `app/pages/*.tsx` 等の機能を、レシピとして書き出して他人に渡すことができます。

### 4.1 出力フロー

1. 「インストール済みレシピ」画面右上の「+ 自作機能をレシピ化」
2. 出力対象の `app/pages/*.tsx`・スタイル・メニュー登録を選択
3. BE 呼び出しが必要な場合、使う handler と scope を宣言
4. メタ情報（レシピ名・説明・作者）を入力
5. 「書き出し」で `.zip` または単体 MD を生成

### 4.2 出力対象の自動検出

KB は `app/menu.ts` のエントリ単位で成果物を関連付けます。「この画面に関係するファイル一式」を自動で候補として提示するので、ほとんどの場合は選択だけで済みます。

### 4.3 出力時の注意

- `app/api/*.ts`（ユーザー定義のバックエンド API）は **レシピには含められません**。レシピは宣言的 handler モデルで動くため、任意コードを含むルートは持ち運べない設計です
- 外部に渡す前に「機密データが含まれていないか」自動チェックが入ります（`.env` 参照や個人情報パターン等）

#### 4.3.1 `api/` を含むアプリのエクスポートは拒否される

エクスポート対象のアプリが `app/<appId>/api/` 配下にカスタムバックエンドコードを含む場合、KB はエクスポートを **拒否**します（HTTP 400 `CustomBeNotExportable`）。レスポンスには検出されたファイルの一覧が含まれます。

このとき UI には **action-first メッセージ**（先に取るべき対応を提示する誘導文）が表示されます。要旨:

> このアプリはレシピとして書き出せません。配布するには次のどちらかの対応が必要です:
> 1. サーバー側の処理を、レシピが扱える宣言的な API 呼び出し（`recipe.yaml` の `api.calls` + `window.kb.call`）に書き直す
> 2. サーバー側の処理を手順書として別途まとめ、レシピのインストール後に利用者がエージェントの助けを借りて実装できるようにする
>
> 理由: `app/<appId>/api/` 配下のコードは、レシピが安全に取り込める範囲の外にあり、レシピに同梱できません。

エージェントは利用者に対し、この 2 択（書き直す / 手順書化する）を起点に案内してください。「`api/` を消せばよい」といった破壊的な近道は提案しないでください。

## §5 公式レシピの案内（同梱サンプル）

KB には以下の公式サンプルレシピがリポジトリ内 `recipes/` に同梱されています。**外部レシピのインストールが無効な v0.2.x でも、これらのサンプルは「アプリ」画面の Sample apps タブから有効化して利用できます。**

### 5.1 ドキュメントビュアー

- **ID:** `document-viewer`
- **何ができるか:** `docs/` 配下や任意のディレクトリの Markdown を、KB 内のビュアーで綺麗に閲覧
- **要求 scope:** `project-read`
- **用途例:** プロジェクトドキュメントを社内共有せずに手元で確認したい

### 5.2 TODO 管理

- **ID:** `todo`
- **何ができるか:** 簡易的なチェックリスト管理。レシピ専用 KV に保存
- **要求 scope:** `own-data`
- **用途例:** 作業メモ・買い物リスト等、プロジェクト外で独立管理したい TODO

### 5.3 サンプルレシピの使い方

「アプリ」画面 → **Sample apps タブ** → 該当サンプルの「有効化」ボタンをクリック。有効化するとアプリとして `app/<appId>/` に配置され、サイドバーのメニューに追加されます。不要になったら同タブから無効化でき、その際 `app/data/<appId>/` 配下のデータは保持されます（再有効化で復帰可能、[`05-apps.md`](./05-apps.md) §10 参照）。

## §6 宣言的 handler モデル（概要）

レシピから BE（サーバー側）機能を使うときは、**レシピがコードを書くのではなく、KB コアが提供する事前定義の handler を呼ぶ宣言だけを書く** 仕組みを採用しています。

### 6.1 なぜこの仕組みか

- **セキュリティ:** コード実行そのものが発生しないため、悪意あるレシピが任意処理を仕込むことが構造的に不可能
- **可視化:** レシピが「何にアクセスするか」が YAML 宣言として読めばわかる
- **クロスプラットフォーム:** Node.js のみで完結

### 6.2 v0.1.0 で提供される handler（Category A、9 個）

| handler 名 | 用途 |
|---|---|
| `list-files` | ディレクトリのファイル一覧取得 |
| `read-file` | ファイル読み取り |
| `write-file` | ファイル書き込み |
| `kv-get` | KV ストアから値取得 |
| `kv-set` | KV ストアへ値保存 |
| `kv-list` | KV ストアのキー一覧 |
| `kv-delete` | KV ストアから値削除 |
| `notify` | デスクトップ通知 |
| `export-file` | ユーザーが指定したパスへのエクスポート（保存ダイアログ経由） |

### 6.3 レシピ YAML での宣言例

```yaml
recipeId: "document-viewer"   # 必須（§1.1 参照）
name: "ドキュメントビュアー"
# ... 他のメタデータ

api:
  scopes:
    - project-read
    - own-data
  handlers:
    - name: read-file
      args:
        path: docs/index.md
    - name: kv-set
      args:
        key: last-viewed
        value: docs/index.md
```

レシピ側はこの宣言しか持てず、「宣言していない handler を叩く」経路はそもそも存在しません。

### 6.4 提供されていない機能

v0.1.0 で提供されない機能（Category B/C）:

- HTTP 外部通信
- シェル実行
- npm パッケージの動的実行
- KB コアの書き換え

これらが必要な場合は、レシピではなく [`05-apps.md`](./05-apps.md) の `app/api/` 側で自分で実装してください（フル権限・自己責任）。

## §7 scope と承認

### 7.1 scope の種類

レシピが要求できる権限は以下の 7 種類です:

| scope | 許可する対象 | 書き込み |
|---|---|---|
| `project-read` | プロジェクトルート配下（除外あり） | 不可 |
| `project-write` | 同上 | 可 |
| `agents-read` | `.claude/agents/` 配下 | 不可 |
| `skills-read` | `.claude/skills/` 配下 | 不可 |
| `claude-md-read` | 各種 `CLAUDE.md` ファイル | 不可 |
| `kb-data-read` | `kovitoboard/data/` 配下 | 不可 |
| `own-data` | `app/data/{appId}/`（このアプリ専用領域） | 可 |

### 7.2 常に拒否される対象（ハードコード除外）

scope に関わらず、以下は **常にアクセス拒否** されます:

- `.env` / `.env.*`
- `.git/` 配下
- `node_modules/` 配下
- `.claude/credentials*`

### 7.3 承認 UX

scope 承認は **エージェントとの対話** で実施します。

フロー:

1. レシピのインストールボタンを押す
2. （declarative の枠を超える実装が含まれている場合のみ）警告ダイアログ表示
3. エージェント選択モーダル
4. **エージェントとの対話で scope の意味を確認** — エージェントがユーザーに自然言語で説明:
   - 「このレシピは `project-read` と `own-data` を使います。プロジェクト内のファイルの読み取りと、このアプリ専用のデータ保存領域へのアクセスです」
   - 「この権限で問題ないですか？」
5. ユーザーが OK を出すと、エージェントがファイル配置を進める

**メリット:**

- ユーザーがエージェントに質問できる（「`project-read` で具体的に何を読みますか？」等）
- エージェントが artifacts のコードを Read してから説明できるため、宣言と実装の乖離をユーザーに伝えられる

**KB が保証する範囲:**

- 承認結果（採択された scope セット）はレシピ manifest に記録され、ランタイムで強制されます
- エージェントが scope を勝手に拡張することはできません（dispatcher は manifest を権限境界として使います）

**後から追加承認は v0.1.0 ではできません**（v0.2.0 以降で検討）。新しい scope が必要なレシピは、更新版を受け取って再インストールしてください。

### 7.4 判断の目安（ユーザー向け）

- `own-data` のみ: ほぼノーリスク（レシピ専用領域で完結）
- `project-read` / 各 `*-read`: 読み取りのみ。漏洩リスクは気にする価値があるが破壊はしない
- `project-write`: 書き込み権限あり。信頼できる配布元かを確認してから承認
- 不明な scope の要求: 拒否して、レシピ作者に説明を求める

### 7.5 監査ログ

レシピが handler を呼んだ履歴は `app/data/{appId}/_audit.log` に自動記録されます。何時にどの handler を何のパスで呼んだかを後から確認できます。

## §8 レシピのトラブルシュート

### 8.1 外部レシピをインストールしたいができない

- **これは仕様です（不具合ではありません）。** v0.2.x では外部レシピのインストール経路が無効です（`/api/recipes/install` / `/api/recipes/apply` とも 410 Gone）。v0.3.0 で KovitoHub と連動して提供予定です（§2）
- 今すぐ試したい場合は、同梱サンプルレシピを「アプリ」画面の **Sample apps タブ**から有効化してください（§5）

### 8.2 サンプルレシピを有効化したのに動かない

- **scope 不足:** 実行時に `ScopeViolation` エラーが出ている可能性。監査ログを確認してください
- **handler 呼び出しの引数ミス:** ファイルパスが scope の範囲外だと拒否されます
- **ブラウザに新しいメニューが出ない:** 有効化直後は **ブラウザのページを再読み込み** してください（サーバー再起動は不要 — `app/menu.ts` の変更が FE 側に反映されるため）

### 8.3 インストールしたレシピを取り消したい

レシピが作ったアプリの削除は **アプリ画面の削除ボタン**（[`05-apps.md`](./05-apps.md) §10 参照）から実施します。レシピ画面に「アンインストール」ボタンはありません。

詳細は §3.3.2 「アプリの削除」を参照してください。

### 8.4 信頼できないレシピを渡されたら

- v0.2.x では外部レシピのインストール経路が無効なため、受け取ったレシピファイルをそのまま実行することはできません（§2）。ファイルを削除するだけで影響はありません
- v0.3.0 の KovitoHub 連動では、署名済み publisher モデル + scope 開示によって信頼性を担保する設計です。不安な scope 要求があれば承認しない、という原則は将来も同じです
