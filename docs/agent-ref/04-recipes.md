## §1 レシピとは

レシピは、KB に **機能の塊（画面 + バックエンド呼び出しの宣言）を追加できる、配布・受け取り可能な単位** です。ドキュメントビュアーや TODO 管理のような「1 つの目的を持ったミニアプリ」を、他のユーザーや Kovito 公式から受け取り、ワンクリックでインストールできます。

### 1.1 レシピの構造（ユーザーから見た視点）

レシピには以下が含まれます:

- **成果物:** 実際に追加される UI ページ・スタイル（`app/pages/*.tsx` 等）
- **メニュー登録:** サイドバーのどこに表示するか
- **BE 呼び出しの宣言（任意）:** ファイル読み取り・KV 保存等の「どの handler を呼ぶか」を YAML で宣言（§6 参照）
- **scope（権限）の宣言:** 何にアクセスしてよいか（§7 参照）
- **指示（任意）:** エージェントへの補足説明

### 1.2 レシピとアプリ（`app/`）の違い

| 項目 | レシピ | ユーザー定義のバックエンド API（`app/api/`） |
|---|---|---|
| 実行経路 | 宣言的 handler モデル経由（サンドボックス） | Node.js でフル権限実行 |
| 権限 | `scope` で宣言した範囲のみ | 制限なし |
| 信頼モデル | 「第三者から受け取っても安全に試せる」 | 「自分で書いたコード、自己責任」 |
| コード実行 | レシピは **コードを書かず宣言するだけ** | TypeScript を自由に書く |
| ホットロード | インストール即時 | サーバー再起動が必要 |

**判断基準:**

- 「他人に渡したい」「受け取りたい」 → **レシピ**
- 「自分専用の作り込んだ機能が欲しい」「フル権限で何でもしたい」 → [`05-apps.md`](./05-apps.md) の **`app/`** 配下で作る

### 1.3 v0.1.0 でのスコープ

- レシピの FE 拡張（`page` 型の成果物）は正式サポート
- BE 拡張は **宣言的 handler モデル** で Category A（9 個、ファイル読み書き・KV・通知・エクスポート）のみ利用可能
- HTTP 外部通信・シェル実行等は v0.1.0 では対応しない（v0.2.0+ で別判断）

## §2 外部レシピの読み込み

### 2.1 レシピ一覧画面の構成

画面左サイドバーの「レシピ」を開くと、2 セクションに分かれています:

- **インストール済みレシピ:** 現在 KB に適用されているレシピ
- **インストール前レシピ:** 受け取ったけどまだ入れていないレシピ（スキャン対象の候補）

### 2.2 受け取り経路（v0.1.0）

v0.1.0 では以下の経路でレシピを受け取れます:

1. **ローカルファイル:** `.zip` または `.yaml`（単体 MD 形式）をドラッグ＆ドロップ
2. **URL:** `https://...` のレシピ URL を貼り付け（起動時スキャナが取得）
3. **リポジトリ内レシピ:** KB 同梱の `recipes/` ディレクトリから選ぶ（§5）

### 2.3 インストールフロー（概要）

外部レシピは以下の流れでインストールされます:

```
受け取り → 静的検査 → 確認画面（scope・コード全開示） → 承認 → 適用 → 履歴記録
```

1. **静的検査:** 禁止された型・危険なパターンがないかチェック
2. **確認画面:** レシピが何をするか・どの権限を要求するか・全コードをユーザーに開示
3. **承認:** 「承認してインストール」ボタンで確定
4. **適用:** `app/pages/` や `app/menu.ts` に展開
5. **履歴記録:** 誰からいつ受け取ったかを `.kovitoboard/recipe-history.json` に記録

### 2.4 起動時スキャナ

KB 起動時、`.kovitoboard/incoming/` 配下に未インストールのレシピがないかスキャンします。見つかると「新しいレシピが届いています」と通知されます（自動インストールはしません）。

## §3 レシピの実行と履歴

### 3.1 レシピの実行

インストール済みレシピは、レシピが追加したメニュー項目（§1.1）から使います。レシピ固有の画面が開き、そこで操作します。

### 3.2 履歴の確認

「インストール済みレシピ」の各カードから、以下が確認できます:

- **インストール日時・経路**（ファイル / URL / 同梱）
- **承認済み scope**
- **実行履歴**（いつどの handler を何回呼んだか、`app/data/{recipe-id}/_audit.log` に記録）

### 3.3 レシピのアンインストール

カードの「アンインストール」ボタンから実行できます。

アンインストール時の挙動:

- `app/pages/` 等の成果物は削除
- `app/menu.ts` からエントリが消える
- `app/data/{recipe-id}/` の **専用データ領域は保持**（再インストール時に復元可能）

データも消したい場合は、「データも含めて削除」を選んでください。

## §4 自作機能のレシピ出力

KB 上で自分が作った `app/pages/*.tsx` 等の機能を、レシピとして書き出して他人に渡すことができます。

### 4.1 出力フロー

1. 「インストール済みレシピ」画面右上の「+ 自作機能をレシピ化」
2. 出力対象の `app/pages/*.tsx`・スタイル・メニュー登録を選択
3. BE 呼び出しが必要な場合、使う handler と scope を宣言
4. メタ情報（レシピ名・説明・作者）を入力
5. 「書き出し」で `.zip` または単体 MD を生成

### 4.2 出力対象の自動検出

KB は `app/menu.ts` のエントリ単位で成果物を関連付けます。「この画面に関係するファイル一式」を自動で候補として提示するので、ほとんどの場合は選択だけで済みます。

### 4.3 出力時の注意

- `app/api/*.ts`（ユーザー定義のバックエンド API）は **レシピには含められません**。レシピは宣言的 handler モデルで動くため、任意コードを含むルートは持ち運べない設計です
- 外部に渡す前に「機密データが含まれていないか」自動チェックが入ります（`.env` 参照や個人情報パターン等）

## §5 公式レシピの案内（v0.1.0 同梱予定）

v0.1.0 では以下の公式レシピがリポジトリ内 `recipes/` に同梱されます。「インストール前レシピ」から即座に入れられます。

### 5.1 ドキュメントビュアー

- **ID:** `document-viewer`
- **何ができるか:** `docs/` 配下や任意のディレクトリの Markdown を、KB 内のビュアーで綺麗に閲覧
- **要求 scope:** `project-read`
- **用途例:** プロジェクトドキュメントを社内共有せずに手元で確認したい

### 5.2 TODO 管理

- **ID:** `todo`
- **何ができるか:** 簡易的なチェックリスト管理。レシピ専用 KV に保存
- **要求 scope:** `own-data`
- **用途例:** 作業メモ・買い物リスト等、プロジェクト外で独立管理したい TODO

### 5.3 同梱レシピの使い方

画面左「レシピ」→「インストール前」タブ → 該当レシピをクリック → 確認画面で承認。

## §6 宣言的 handler モデル（概要）

レシピから BE（サーバー側）機能を使うときは、**レシピがコードを書くのではなく、KB コアが提供する事前定義の handler を呼ぶ宣言だけを書く** 仕組みを採用しています（DEC-006）。

### 6.1 なぜこの仕組みか

- **セキュリティ:** コード実行そのものが発生しないため、悪意あるレシピが任意処理を仕込むことが構造的に不可能
- **可視化:** レシピが「何にアクセスするか」が YAML 宣言として読めばわかる
- **クロスプラットフォーム:** Node.js のみで完結

### 6.2 v0.1.0 で提供される handler（Category A、9 個）

| handler 名 | 用途 |
|---|---|
| `list-files` | ディレクトリのファイル一覧取得 |
| `read-file` | ファイル読み取り |
| `write-file` | ファイル書き込み |
| `kv-get` | KV ストアから値取得 |
| `kv-set` | KV ストアへ値保存 |
| `kv-list` | KV ストアのキー一覧 |
| `kv-delete` | KV ストアから値削除 |
| `notify` | デスクトップ通知 |
| `export-file` | ユーザーが指定したパスへのエクスポート（保存ダイアログ経由） |

### 6.3 レシピ YAML での宣言例

```yaml
api:
  scopes:
    - project-read
    - own-data
  handlers:
    - name: read-file
      args:
        path: docs/index.md
    - name: kv-set
      args:
        key: last-viewed
        value: docs/index.md
```

レシピ側はこの宣言しか持てず、「宣言していない handler を叩く」経路はそもそも存在しません。

### 6.4 提供されていない機能

v0.1.0 で提供されない機能（Category B/C）:

- HTTP 外部通信
- シェル実行
- npm パッケージの動的実行
- KB コアの書き換え

これらが必要な場合は、レシピではなく [`05-apps.md`](./05-apps.md) の `app/api/` 側で自分で実装してください（フル権限・自己責任）。

## §7 scope と承認

### 7.1 scope の種類

レシピが要求できる権限は以下の 7 種類です:

| scope | 許可する対象 | 書き込み |
|---|---|---|
| `project-read` | プロジェクトルート配下（除外あり） | 不可 |
| `project-write` | 同上 | 可 |
| `agents-read` | `.claude/agents/` 配下 | 不可 |
| `skills-read` | `.claude/skills/` 配下 | 不可 |
| `claude-md-read` | 各種 `CLAUDE.md` ファイル | 不可 |
| `kb-data-read` | `kovitoboard/data/` 配下 | 不可 |
| `own-data` | `app/data/{recipe-id}/`（このレシピ専用領域） | 可 |

### 7.2 常に拒否される対象（ハードコード除外）

scope に関わらず、以下は **常にアクセス拒否** されます:

- `.env` / `.env.*`
- `.git/` 配下
- `node_modules/` 配下
- `.claude/credentials*`

### 7.3 承認 UX

インストール時、以下のようなダイアログが出ます:

```
このレシピは以下の権限を要求しています:
  📖 プロジェクトファイルの読み取り（project-read）
  💾 レシピ専用データ領域（own-data）

[承認してインストール]  [拒否]
```

- **承認すると:** インストール後、レシピはこの範囲内でのみ動作
- **後から追加承認は v0.1.0 ではできません**（v0.2.0 以降で検討）。新しい scope が必要なレシピは、更新版を受け取って再インストールしてください

### 7.4 判断の目安（ユーザー向け）

- `own-data` のみ: ほぼノーリスク（レシピ専用領域で完結）
- `project-read` / 各 `*-read`: 読み取りのみ。漏洩リスクは気にする価値があるが破壊はしない
- `project-write`: 書き込み権限あり。信頼できる配布元かを確認してから承認
- 不明な scope の要求: 拒否して、レシピ作者に説明を求める

### 7.5 監査ログ

レシピが handler を呼んだ履歴は `app/data/{recipe-id}/_audit.log` に自動記録されます。何時にどの handler を何のパスで呼んだかを後から確認できます。

## §8 レシピのトラブルシュート

### 8.1 レシピが読み込めない

- **ファイル形式が対応していない可能性:** `.zip` または `.yaml` のみ対応（v0.1.0）
- **静的検査で弾かれた可能性:** 禁止された型（`api` 以外の非対応成果物タイプ等）が含まれていると読み込みが止まります。確認画面のエラーメッセージを見てください
- **URL から取得に失敗:** ネットワーク接続を確認してください

### 8.2 インストール後にレシピが動かない

- **scope 不足:** 実行時に `ScopeViolation` エラーが出ている可能性。監査ログを確認してください
- **handler 呼び出しの引数ミス:** ファイルパスが scope の範囲外だと拒否されます
- **ホットロードが効いていない:** KB を再起動してみてください

### 8.3 インストールしたレシピを取り消したい

§3.3 のアンインストール手順を参照してください。

### 8.4 信頼できないレシピを受け取ったら

- 確認画面で scope・全コードが開示されます。不安な要求があれば **拒否** してください
- ローカルファイル形式で受け取った場合、ファイルを削除するだけで影響なし（インストール前なら実害ゼロ）
