## §1 スキル管理（読み取り専用の参照）

### 1.1 スキルとは

ここでのスキルは **Claude Code のスキル機能** を指します。特定タスクに特化した手順書＋挙動定義で、`.claude/skills/` 配下に `*.md` として配置されます。

### 1.2 KB からの扱い

- v0.1.0 では **読み取り中心**。KB の UI からスキル一覧は見えますが、編集は外部（エディタ or Claude Code CLI）で行うのが推奨です
- エージェント（特にコビー）はスキル定義を参照して回答の質を上げる場合がありますが、KB 側でスキルを自動追加・改変することはしません

### 1.3 注意

- ユーザーがインストールした不明なスキルが悪意を持つ可能性（「ClawHavoc」型の流通リスク）があります。KB 自体はスキル実行をサンドボックス化しません
- 信頼できない配布元のスキルを `.claude/skills/` に入れないでください（レシピと違って scope 制御はありません）

## §2 自動処理・ルール設定の閲覧

### 2.1 Claude Code の Hook

`.claude/settings.json` の hooks で、特定イベント時に自動コマンドが走るように設定できます（Pre/Post tool use・Stop 等）。KB はこれを **読むだけ** で、UI からの編集は提供しません。

### 2.2 KB 固有のルール

v0.1.0 時点では、KB 固有の自動ルール機構は提供しません。自動化が必要なら:

- Claude Code 側の hooks を設定する
- `app/api/` でエンドポイントを作り、cron / systemd timer 等で叩く

### 2.3 今後の展望

v0.2.0 以降で、KB 側に「スケジュール実行」「イベントトリガ」等を検討する可能性があります。v0.1.0 では対応範囲外です。

## §3 `.kovitoboard/` 配下の構造

`.kovitoboard/` はプロジェクトルート直下に作られる KB のデータ・設定領域です。通常ユーザーが直接触る必要はありませんが、構造を知っておくとトラブルシュートや応用に役立ちます。

### 3.1 主要な配置

```
.kovitoboard/
├── setting.json           ← マスタ情報（userName, projectName 等）
├── assets/
│   └── agents/
│       └── {agent-id}/
│           ├── custom.png
│           └── default.png
├── incoming/              ← 起動時スキャン対象のレシピ受け取り場所
├── recipe-history.json    ← レシピのインストール履歴
└── logs/                  ← KB 自身のログ
```

### 3.2 直接触ってもよい場所

- `setting.json`: UI からの変更が推奨だが、JSON 構文を崩さなければ直編集可
- `incoming/`: レシピの `.zip` / `.yaml` を配置すると起動時に検出される

### 3.3 触らないほうがよい場所

- `assets/` 配下: UI 経由のアップロードで配置する
- `recipe-history.json`: インストール履歴の整合性が壊れる

## §4 ログ・デバッグ情報の確認

### 4.1 KB 本体のログ

`.kovitoboard/logs/` 配下にイベントログが追記されます（起動・エージェント追加・レシピインストール等）。トラブル発生時はまずここを見てください。

### 4.2 Claude Code のログ

`.claude/logs/` 配下に Claude Code 側のログ（セッション出力・tool use 記録）が残ります。セッション単位でディレクトリが分かれます。

### 4.3 レシピの監査ログ

レシピが handler を呼んだ履歴は `app/data/{recipe-id}/_audit.log` に記録されます。`(timestamp, handler, args 概要, 結果)` が追記式で残ります。

### 4.4 ログの取り扱い

- ログには機密情報が混ざる可能性があります。第三者と共有する前に内容を確認してください
- 長期的な肥大化を避けたい場合は、定期的に古いログをアーカイブ or 削除してください（KB 側の自動ローテーションは v0.1.0 では未実装）

## §5 複数プロジェクトでの利用

### 5.1 基本方針

KB は **プロジェクトごとに 1 インスタンス** を前提に設計されています。同一マシン上で複数プロジェクトを並行運用するには、以下のいずれか:

1. **別ポートで並行起動:** supervisor (`tools/kb-start.mjs`) は起動時に空きポートを自動探索するため、デフォルト構成のまま 2 つ目以降を `npm start` するだけで、Backend は 3002 / 3003 ...、Frontend (Vite) は 5174 / 5175 ... のように **自動的に別ポートで立ち上がります**。supervisor の起動ログ末尾に表示される `[kb-start] Frontend: http://localhost:<port>` の URL を各々開いてください。明示的にポートを固定したい場合は `npm start -- --port=8080 --vite-port=8000` のように指定できます（指定ポートが使用中ならエラーで終了）
2. **順次切替:** 1 プロジェクトずつ起動する

### 5.2 `.kovitoboard/` の分離

各プロジェクトの `.kovitoboard/` はプロジェクト配下に独立して作られます。ユーザー名等のマスタ情報は **プロジェクトごとに別個** に管理されます（グローバルに共有したいニーズは v0.2.0+ で検討）。

### 5.3 エージェントの共有

`.claude/agents/` も同様にプロジェクトごとに独立します。共通で使いたいエージェントはプロジェクト間でコピーするか、シンボリックリンクを張って共有してください。

## §6 v0.1.0 で未対応の領域

参考として、v0.1.0 ではサポートしない上級機能を挙げておきます:

- Category B/C handler（HTTP 外部通信・シェル実行等）
- レシピの段階的 scope 承認
- KB 固有のスケジュール実行機構
- マルチプロジェクト連携（同時起動・データ共有）
- `.kovitoboard/` の自動マイグレーション（スキーマ進化時）

これらは v0.2.0 以降で検討します。現時点で必要な場合は、`app/api/` 側で自前実装するか、外部ツールと組み合わせてください。
