## §1 起動しない

### 1.1 症状

- KB のコマンドを実行したが画面が開かない
- ブラウザでアクセスしても真っ白・エラー（デフォルトは `localhost:5173` ですが、後述のとおりポート競合時は別ポートに自動フォールバックします）
- ターミナルにエラーが出て止まる

### 1.2 対処手順

1. **Node.js のバージョンを確認:** v0.1.0 は Node.js 20 以降を前提とします。`node -v` で確認してください
2. **依存が入っているか確認:** `npm install` を実行していないと起動できません
3. **接続先 URL を確認:** `npm start` 実行時の supervisor 出力に `[kb-start] Frontend: http://localhost:<port>  ← open this in your browser` という行が出ます。デフォルト 5173 が他プロセスで使われていた場合は 5174 / 5175 ... に自動でフォールバックするため、必ずこの行の URL を開いてください
4. **ポート衝突を意図的に避ける:** 自動フォールバックを使わずポートを固定したい場合は `npm start -- --port=8080 --vite-port=8000` のように CLI フラグを指定できます。指定したポートが使用中なら supervisor がエラーで終了します
5. **過去のプロセスが残っていないか:** `tmux list-sessions` で古い KB セッションが残っていたら `tmux kill-server` してから再起動
6. **ログを確認:** ターミナル出力の最後のエラーメッセージをユーザーに伝えて一緒に判断

### 1.3 どうしても直らないとき

KB リポジトリ内の `README.md` のトラブルシュート節、または GitHub Issues を確認してください。

## §2 エージェントが表示されない

### 2.1 症状

- エージェント一覧画面にカードが出ない
- 追加したはずのエージェントが出てこない

### 2.2 原因候補と対処

1. **ファイル名と `name` の不一致:** `.claude/agents/` 配下のファイル名（拡張子を除く）と frontmatter の `name` が一致している必要があります
2. **frontmatter が壊れている:** YAML の構文エラーがあるとスキップされます。`---` で挟まれた部分を確認
3. **KB を再読込していない:** ファイル直編集の場合は KB の再起動が必要です
4. **画像が出ないだけで実在する:** アバターはフォールバック生成されるので「画像が出ない = 消えた」ではありません。カードのタイトル領域を確認してください

詳細は [`02-agents.md`](./02-agents.md) を参照してください。

### 2.3 画像が出ない場合

- フォールバック生成は常に動きます（`name` からハッシュ生成したアバター）
- もしフォールバックも出ていない場合、エージェント自体が認識されていません。§2.2 を確認

## §3 セッションが進まない

### 3.1 症状

- メッセージを送ったが応答が返ってこない
- 「応答中...」の表示が消えない

### 3.2 対処手順

1. **Esc キーで割り込み:** エージェントが長考しているだけの可能性があります。一度止めてから短い質問を投げて応答が返るか確認
2. **tmux ペインを確認:** `tmux list-panes` で対象ペインが生きているか
3. **Claude Code のログ確認:** セッションのログに API エラー等が出ていないか（`.claude/logs/` 配下）
4. **セッションを終了して再作成:** 最も確実な復旧策。ただし現セッションのコンテキストは失われます

### 3.3 断続的に固まる場合

- ネットワーク接続を確認（Claude API へのアクセスがタイムアウトしている可能性）
- 短時間に大量のメッセージを送っていないか（レート制限）

## §4 レシピが入れられない / 動かない

> ⚠️ **v0.2.x の前提:** 外部レシピのインストール経路は v0.2.x では無効です（`/api/recipes/install` / `/api/recipes/apply` は 410 Gone）。`.zip` / `.yaml` / URL からの外部レシピのインストールはできません。v0.3.0 で KovitoHub と連動して提供予定です。今すぐ試したい場合は同梱サンプルレシピを使ってください（[`04-recipes.md`](./04-recipes.md) §2 / §5）。

### 4.1 「外部レシピをインストールできない」

- これは仕様です（不具合ではありません）。外部レシピのインストール経路は v0.2.x では無効です（上記）
- 同梱サンプルレシピ（Document Viewer / TODO）は「アプリ」画面の **Sample apps タブ**から有効化できます（[`04-recipes.md`](./04-recipes.md) §5）

### 4.3 有効化したサンプルレシピが動かない場合

- **scope 不足:** `ScopeViolation` エラーが監査ログ（`app/data/{recipe-id}/_audit.log`）に出ている可能性
- **handler 引数エラー:** パスが scope の範囲外だと handler が拒否されます

詳細は [`04-recipes.md`](./04-recipes.md) §8 を参照してください。

## §5 ファイルアクセスエラー

### 5.1 症状

- レシピ実行時に「ファイルが見つかりません」「アクセスが拒否されました」
- `app/api/*.ts` 内で `readFileSync` がエラー

### 5.2 原因候補と対処

1. **パスの基準がずれている:** KB のプロセスは KB リポジトリルートを cwd として動きます。相対パスはルートから解釈してください
2. **scope 範囲外の指定（レシピ）:** レシピから触れるのは scope で宣言した範囲のみ。`.env` や `.git/` は scope に関わらず常に拒否
3. **OS の権限不足:** ユーザーがファイルシステム的に読めないパスは KB も読めません
4. **Windows の path separator:** バックスラッシュとスラッシュの混在に注意（`path.join` を使えば解決）

### 5.3 どこを触れるかの整理

| 区分 | 触れる範囲 |
|---|---|
| レシピ（`project-read`） | プロジェクトルート配下（`.env*`・`.git/`・`node_modules/`・`.claude/credentials*` を除く） |
| レシピ（`own-data`） | `app/data/{recipe-id}/` |
| `app/api/` | Node.js 全域（OS 権限が許す限り） |

## §6 Claude Code が応答しない

### 6.1 症状

- KB 経由で Claude Code を呼ぼうとしても反応がない
- タイムアウトエラーが出続ける

### 6.2 原因候補と対処

1. **Claude Code CLI の認証切れ:** ターミナルで `claude` を直接実行してログイン状態を確認
2. **CLAUDE.md の構文エラー:** KB 起動時に読み込む `.claude/CLAUDE.md` や `CLAUDE.md` に構文エラーがあると止まることがあります
3. **サブエージェント定義の frontmatter 不正:** §2.2 と同じ。YAML が壊れていると対象エージェントが呼べません
4. **Claude API 側の障害:** Anthropic status ページを確認

### 6.3 プロセス系のリカバリ

- `claude` プロセスが残っている場合: `ps aux | grep claude` → 該当 PID を `kill`
- それでも直らない場合は OS 再起動を検討

## §7 trust prompt（信頼確認）が繰り返し出る

### 7.1 症状

- Claude Code が新しいファイル・スキル・ディレクトリを使うたびに信頼確認ダイアログが出る
- 一度承認したはずなのにまた出る

### 7.2 原因と仕組み

Claude Code は、未知のディレクトリ（`.claude/` 配下含む）の実行前に信頼確認を出します。これはセキュリティ機能で、KB が勝手に抑制できるものではありません。

### 7.3 対処

1. **ディレクトリ単位で承認する:** 一度の承認で当該ディレクトリ配下が一括信頼されます
2. **繰り返し出る理由が不明なとき:** Claude Code の trust データベースが壊れている可能性。`.claude/` 配下の trust ファイルを確認（直編集は慎重に）
3. **新しいレシピをインストールした直後:** レシピが追加したファイル群に trust prompt が出るのは正常。承認してください

### 7.4 KB の立ち位置

KB はユーザーに代わって trust を勝手に承認しません。**ユーザー自身が判断する** ことが仕様です。エージェント（コビー含む）も、ユーザーに承認を強く促さず、必要性を説明してから判断を仰いでください。

## §8 バージョン関連の警告 / アップデート時のエラー

### 8.1 ヘッダーに「Claude Code 範囲外」の警告が出ている

- **意味:** インストール済み Claude Code のバージョンが KB の動作確認範囲外（`bestEffortVersions` 外、ベストエフォート対応のみのティア）
- **対処:**
  1. ヘッダーのバッジ → popover で表示されている `primary` バージョンを確認
  2. ターミナルで `npm install -g @anthropic-ai/claude-code@stable` を実行
  3. KB を再起動
- **動作はしているが警告が消えない:** `claude --version` の出力フォーマットが解析できていない可能性。ログを確認

### 8.2 ヘッダーに「KB の新版が公開されています」が出ている

- **意味:** GitHub Releases に新しい KB のバージョンが publish されている
- **対処（推奨）:** popover の「アップデートする」ボタン → `kovito-developer` エージェントが手順を案内
- **対処（手動）:** [`10-upgrade.md`](./10-upgrade.md) §3「バージョンアップ手順（標準フロー）」を参照
- **警告が消えない:** KB 再起動後も残る場合、popover 内「いまチェック」ボタンを押してキャッシュを無視した再取得を試す

### 8.3 アップデート時に conflict が発生した

- **症状:** `git pull` 実行時に `CONFLICT (content): Merge conflict in <path>` が出る
- **対処:** [`10-upgrade.md`](./10-upgrade.md) §4「conflict 発生時の対処」を参照
- **要警戒の典型:** `src/` 配下のファイルで conflict が出ている場合、ユーザーが過去にコアを直接改変している可能性（`05-apps.md` §1.1.1）

### 8.4 アップデート後に動かなくなった

- **対処順:**
  1. `.kovitoboard/logs/server.log` の直近エラーを確認
  2. ブラウザコンソール（F12）の最初のエラーを確認
  3. それでも特定できない場合は [`10-upgrade.md`](./10-upgrade.md) §6「rollback」で戻す
- **典型原因:** npm 依存の更新が反映されていない（`npm install` の実行漏れ）/ `app/` 内のコードが新しいコア API に追従していない

### 8.5 popover の version 情報が「取得できませんでした」のまま

- **原因候補:**
  - ネットワーク非接続 / 企業 firewall で GitHub API がブロック
  - GitHub API レート制限（未認証 60 req/h/IP）
  - `versionCheckEnabled: false` に設定されている（`~/.kovitoboard/config.json`）
- **対処:** ネットワーク状況を確認 / 設定を確認 / 時間をおいて popover の「いまチェック」ボタンを試す。**警告が出ないだけで KB の動作には影響ありません**

---

## §9 それでも解決しない場合

- 症状を整理して「Kovito 開発者」エージェントに相談してください
- ログファイル（`.claude/logs/`・`.kovitoboard/logs/`）の直近エントリがあると診断が進みます
- 再現手順を書き出して GitHub Issues に投稿することも検討してください
